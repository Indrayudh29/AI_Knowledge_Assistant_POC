SOLID Principles
Introduction
SOLID is a set of five object-oriented design principles that help developers build
software that is maintainable, scalable, flexible, and easy to test. These principles
were introduced by Robert C. Martin, commonly known as Uncle Bob, and are
considered the foundation of clean software architecture.
SOLID principles encourage developers to write code that is:
Modern .NET applications, including ASP.NET Core, Entity Framework Core,
Minimal APIs, and Microservices, rely heavily on SOLID principles to achieve clean
architecture and maintainability.

S — Single Responsibility Principle (SRP)
Definition
A class should have only one reason to change.
In other words, a class should have only one responsibility.

O — Open/Closed Principle (OCP)
Definition
Software entities should be open for extension but closed for modification.
This means new functionality should be added without changing existing code.
Bad Example
public class DiscountService
{
public decimal Calculate(string customerType)
{
if(customerType == "Regular")
return 5;
if(customerType == "Premium")
return 10;
return 0;
}
}
Adding a new customer type requires modifying the existing class.
Good Example
public interface IDiscount
{
decimal Calculate();
}
public class RegularCustomer : IDiscount
{
public decimal Calculate() => 5;
}
public class PremiumCustomer : IDiscount
{
public decimal Calculate() => 10;
}
Adding a VIP customer only requires creating another implementation.
public class VipCustomer : IDiscount
{
public decimal Calculate() => 20;
}
No existing code changes.
Easier extension
Lower risk of bugs
Better maintainability
Supports plugin architectures
Benefits of OCP
L — Liskov Substitution Principle (LSP)
Definition
Derived classes should be replaceable with their base classes without
changing the correctness of the program.
Objects of child classes should behave as expected when used through the parent
type.
Bad Example
public class Bird
{
public virtual void Fly()
{
}
}
public class Penguin : Bird
{
public override void Fly()
{
throw new NotImplementedException();
}
}
Penguins cannot fly, violating the expected behavior of Bird.
Good Example
public abstract class Bird
{
}
public interface IFlyingBird
Reliable inheritance
Better polymorphism
Fewer runtime errors
Improved design consistency
{
void Fly();
}
public class Sparrow : Bird, IFlyingBird
{
public void Fly()
{
}
}
public class Penguin : Bird
Only birds that can fly implement IFlyingBird.
Benefits of LSP
I — Interface Segregation Principle (ISP)
Definition
Clients should not be forced to depend on interfaces they do not use.
Large interfaces should be split into smaller, focused interfaces.
Bad Example
public interface IWorker
{
void Work();
void Eat();
void Sleep();
}
A robot worker doesn't need Eat() or Sleep().
Smaller interfaces
Easier implementation
Less unnecessary code
Better flexibility
Abstractions should not depend on details.
Good Example
public interface IWork
{
void Work();
}
public interface IEat
{
void Eat();
}
public interface ISleep
{
void Sleep();
}
Implement only what is needed.
public class Robot : IWork
{
public void Work()
{
}
}
Benefits of ISP

Common Mistakes
Overusing Interfaces
Not every class requires an interface. Creating interfaces for simple, unlikely-to-
change classes can introduce unnecessary complexity.
Large Service Classes
Avoid creating "God Classes" that contain many unrelated responsibilities.

Prefer composition over inheritance where appropriate.
Program against interfaces rather than implementations.
Use Dependency Injection.
Keep interfaces small and focused.
Write unit tests for business logic.
Avoid duplicate code.
Separate business logic from infrastructure concerns.
Refactor large classes into smaller, cohesive components.
Apply SOLID principles consistently rather than selectively.

Interview Questions
1. What does SOLID stand for?
SOLID stands for Single Responsibility, Open/Closed, Liskov Substitution, Interface
Segregation, and Dependency Inversion.
2. Which SOLID principle is most closely related to Dependency Injection?
The Dependency Inversion Principle (DIP). Dependency Injection is a practical
implementation of DIP, allowing classes to depend on abstractions instead of
concrete implementations.
3. Which principle helps avoid "God Classes"?
The Single Responsibility Principle (SRP).
4. Why is the Open/Closed Principle important?
It allows applications to evolve by adding new functionality through extensions rather
than modifying existing, tested code, reducing the risk of introducing bugs.
5. Which SOLID principle improves unit testing the most?
While all principles contribute, the Dependency Inversion Principle (DIP) has the
greatest impact because dependencies can be mocked or substituted during testing.
Summary
SOLID principles provide a foundation for designing robust, maintainable, and
extensible object-oriented software. By ensuring that classes have focused
responsibilities, depend on abstractions, expose only necessary interfaces, and
support safe extension and substitution, developers can build applications that are
easier to understand, test, and evolve. In the .NET ecosystem, SOLID principles work
hand in hand with Dependency Injection, ASP.NET Core, Entity Framework Core, and
Clean Architecture, making them essential knowledge for building enterprise-grade
applications.
